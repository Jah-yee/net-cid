# Lessons

## 2026-02-20

- When a user asks to "fix all findings" from a security report, do not stop at recommendations. Resolve each finding in code/dependencies, re-run security commands, and update the report status to resolved.
- When the user references an external examples guide, inspect that guide directly and mirror its example taxonomy in the local `examples/` directory.

## 2026-05-21

- AGENTS.md "Plan First / Verify Plan" is non-negotiable. Write the plan to `tasks/todo.md` and report it to the user for verification **before** any implementation or file write — even when the issue body looks self-explanatory and the autonomous-bug-fixing principle would otherwise apply. Implementation-first is a correction-worthy violation.

## 2026-06-07

- RFC 8785 §3.2.2.3 mandates ECMA-262 §6.1.6.1.20 (`Number.prototype.toString`), which operates on IEEE-754 doubles. Any "integer fast path" that emits an integer literal verbatim is correct only for |x| ≤ 2^53 (the largest integer exactly representable as a double). Outside that range, the fast path silently violates the spec: a 22-digit integer and the same value re-typed with a `.0` suffix produce different canonical bytes, breaking the determinism guarantee. Gate or drop the fast path; do not assume "integer in CLR range" ⇒ "ECMA-identical output". Reason: caught by adversarial review on issue #13; previously masked by tests that pinned the buggy verbatim output.
- Adversarial review is non-negotiable per AGENTS.md §2 — launch it before declaring a non-trivial change done. The agent caught a high-severity correctness bug after green tests, green build, green example. "All tests pass" is necessary but not sufficient; the test suite can lock in a bug as readily as it can catch one.
- A test named for a stack-overflow / DoS scenario must use input large enough to ACTUALLY trigger the failure mode if the guard were removed — not merely cross the policy threshold. Issue #16: the JsonElement over-limit test used depth 70, which proves the depth-64 gate but would not overflow the stack on its own (the real `WriteElement` recursion overflow threshold is ~2,500 frames, empirically measured by the adversarial probe). A "just-over-limit boundary" test and a "far-over no-crash" test are distinct obligations: the boundary test documents the policy; only the far-over test proves the crash is prevented — and for it, confirm the chosen magnitude would actually crash/hang pre-fix. Note the JsonElement path (no `SerializeToDocument` boundary) overflows far shallower than the JsonNode path. Caught by adversarial review on issue #16.
- When adding a DEFAULT-ON limit to a low-level API, audit every higher-level convenience API that calls it — they silently inherit the new default and may need their own override parameter + doc + changelog note. Issue #16: adding the default 1 MiB output cap to `JcsCanonicalizer.Canonicalize` silently capped `Cid.FromCanonicalJson` (the documented one-call CID-from-JSON path) with no escape hatch, and the breaking-change note only mentioned the `Canonicalize` overloads. Fix added an optional `maxOutputBytes` to `FromCanonicalJson`. Caught by adversarial review on issue #16.
- A default-ON behavior change needs a test that pins the DEFAULT, not just the explicitly-parameterized form. Issue #16: all output-cap tests passed explicit tiny limits, so a regression delegating with `int.MaxValue` (disabling the 1 MiB default) passed the whole suite green; the `Default_…_Is_One_MiB` const-equality test did not catch it. Add a test that exercises the default path end-to-end (canonicalize >1 MiB through the parameterless overload and assert the throw). Caught by adversarial review on issue #16.
- Before planning or implementing a GitHub-issue fix in this repo, `git fetch origin` and reset/rebase onto the latest `origin/main` FIRST — `net-cid` moves fast and adjacent issues routinely rewrite the same files. Issue #17 (JCS duplicate keys) started from a base 15 commits stale: #16 had already added depth (`MaxDepth=64`) and output-size (`LimitedBufferWriter`) guards to `JcsCanonicalizer.cs` and changed every writer signature to thread a `depth` param, and the spec doc had been renamed `net-cid-implemented-specs.md` → `SPECIFICATIONS.md`. Planning against the stale tree would have produced edits that don't apply. The user had to remind me to fetch latest origin/main — bake the sync into step 0 of every issue task, and re-read the target files after rebasing.
- Don't go straight to `main` for an issue fix — create a `feature/issue-N-...` branch BEFORE the first edit (the repo's convention; `git checkout -b` carries already-made uncommitted edits onto the new branch, so it's recoverable, but branch first). The user had to remind me on issue #18 after I'd started editing on `main`.
- An issue's own proposed fix can be wrong — verify the failure mechanism empirically before writing the plan, don't transcribe the issue's patch. Issue #18 proposed a throwing UTF-8 encoder in `WriteString` to reject unpaired surrogates. A throwaway repro proved that catches **nothing** on the real paths: `JsonSerializer.SerializeToDocument`/`SerializeToNode`/`Serialize` substitute an unpaired surrogate with U+FFFD at the SERIALIZE step (confirmed: the document's inner `GetString()` returns a valid U+FFFD), so by the time `WriteString` runs there is no surrogate left to catch; and the `JsonElement` path throws `InvalidOperationException` at `GetString()`/`prop.Name` *before* `WriteString`. The correct fix validates the raw `JsonNode` tree BEFORE serialization (`ValidateNode` → `ThrowIfUnpairedSurrogate` on string values + member names) and converts the parsed-path `InvalidOperationException` → `JcsFormatException`. There is no single downstream choke point because STJ corrupts/throws upstream.
- Validating a `JsonValue` by type-enumeration (`TryGetValue<double>/<float>/<string>`) is incomplete and leaks. Issue #18 adversarial review (after green tests + green example) found a `char`-backed value (`JsonValue.Create('\uD800')`) bypassed the `string`-only check and silently produced U+FFFD — a single UTF-16 code unit is always an unpaired surrogate, so add an explicit `TryGetValue<char>` + `char.IsSurrogate` check. A second round found that ANY `JsonValueCustomized<T>` wrapping a raw CLR object (POCO / `Dictionary` / array via `JsonValue.Create(object)`) also leaks, and CANNOT be fixed: STJ substitutes U+FFFD during serialization before any inspection, and no STJ path preserves the raw surrogate, so reflecting over arbitrary CLR types would be the only option. The parsed-JSON threat model is unaffected (`JsonNode.Parse`/`JsonElement` never produce `JsonValueCustomized<T>`); the residual gap was documented (user's call) rather than fixed by rejecting all object-backed nodes (which would also reject valid surrogate-free POCOs). Lesson: run adversarial review in MORE THAN ONE round on security fixes — the first round's point-fix can reveal a wider class.
